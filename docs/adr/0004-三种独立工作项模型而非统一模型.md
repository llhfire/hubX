# 0004-三种独立工作项模型而非统一模型

项目模块引入需求、任务、缺陷三种工作项时，采用三种独立模型（各自独立字段和状态流），而非统一工作项模型 + type 区分。

## 状态

已接受

## 背景

工作项需要支持需求、任务、缺陷三种类型。它们虽然共享部分字段（标题、负责人、优先级、状态），但各自有独特的字段和状态流：需求有验收标准，任务有预计/实际工时和检查清单，缺陷有严重程度和复现步骤。

## 决策

采用三种独立模型：`Requirement`、`Task`、`Defect`，各自定义独立字段和状态流。评论和操作历史共用一张表（通过 `workItemType` + `workItemId` 关联）。

## 理由

需求/任务/缺陷的业务属性和状态流差异显著：
- 缺陷有"待验证"中间状态（修复后需提交人验证才能关闭），需求和任务没有
- 任务有工时字段（预计/实际），需求和缺陷没有
- 缺陷有"严重程度"（severity）和"优先级"（priority）两个独立维度

硬揉成一个统一模型会让结构变成"胖"字段（大量可选字段），约束力弱，且状态流混乱。

## 后果

- 优势：每种工作项的字段和状态流清晰、强约束，UI 和校验逻辑简单
- 风险：跨类型统计需要 union 查询，但此场景可通过前端聚合解决
