# 上下文

工作台主页重做（2026-07）新增「线索推进漏斗」区块，展示线索在五个阶段的分布：未触达 / 初步沟通 / 跟进中 / 促成阶段 / 已流失。

HubX 现有线索实体已有 `status` 字段，取值：需求调研 / 方案报价 / 合同洽谈 / 初步沟通（见 `src/app/pages/Dashboard.tsx:64-83` 的 mock）。该字段被 `LeadDetail`、`MyLeads`、`PublicLeads` 等多个页面读取。

漏斗需要五阶段语义，与现有 `status` 四值既不数量对齐、也不阶次对齐。

## 决策驱动因素

1. **C 路线约束**：本次工作台重做定位为「视觉与布局先行，业务概念占位、不扩张数据层、不改 CONTEXT 业务语义」。如果改 `status` 取值集合扩展到五值，将污染所有已读 `status` 的现有页面（`LeadDetail` 等），属越界。
2. **现有 mock 一致性**：`status` 现存页面的状态徽标、过滤逻辑都基于四值集合，改值要全栈回归。
3. **占位诚实**：新漏斗字段是纯前端占位，未来接入真实数据时需重新评估与 `status` 的合并策略——这一步决策应延后到接入真实业务时做，而不是现在就强行合并。

# 决策

引入独立占位字段 `leadStage: 'unreached' | 'contacted' | 'following' | 'closing' | 'lost'`，仅存在于 `src/app/pages/dashboard/types.ts`。

- 漏斗区块只读 `leadStage`，**绝不读 `status`**。
- 现有页面继续用 `status`，工作台不触碰。
- 两字段在工作台这一屏并置呈现：漏斗读 `leadStage`、待跟进线索表格读 `status`。
- 漏斗图表标题写「线索推进漏斗」，与表格「待跟进线索」做语义词区分。

`leadStage` 在类型文件上标注 `// TODO: 接入真实数据时评估与 status 合并或并存`。

# 后果

**正向**
- 现有 `LeadDetail` / `MyLeads` / `PublicLeads` 等页面零侵入。
- 漏斗五段语义清晰，不受现有 `status` 四值束缚。
- 未来合并不被锁死——延后到真接业务数据时再决定。

**负向**
- 工作台内部存在两套阶段语义（`status` 表格 + `leadStage` 漏斗），未来读者可能困惑「为什么有两套阶段字段」——本 ADR 即为回答这一困惑。
- `leadStage` 和 `status` 之间的值映射规则尚未定（「需求调研」对应漏斗哪段?），接入真实数据时要补映射函数。

# 替代方案

**A. 扩展现有 `status` 到五值**：合并字段减少双轨。代价是改 `status` 取值集合，污染所有读它的页面，违背 C 路线「不扩张既有数据层」约束。**被否**。

**B. 漏斗复用 `status` 四值绘四段**：不改 `status`，漏斗只画四段。代价是漏斗与参考图五段不符（缺「未触达」「已流失」两段），且现有四值命名顺序不构成严格漏斗递进。**被否**。
