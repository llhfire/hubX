# 站内消息提醒（流程型待办）设计文档

**日期**: 2026-05-21
**状态**: 待评审

---

## 1. 背景与目标

当前系统顶部导航已存在日报未提交提示和静态通知铃铛入口，但提醒能力仍分散在页面局部状态中，尚未形成统一的“待我处理”体系。此次需求来自原始需求池中的“站内消息提醒”，但第一期重点不是建设完整站内信中心，而是把与本人相关的流程型待办事项统一显性化，并引导用户及时处理。

**第一期目标**：
- 统一承载与本人相关的待办提醒，而不是分散在各业务页面各自处理
- 支持日报、审批、客户/合同、线索等核心流程的提醒收敛
- 让用户能从顶部入口快速感知待办，从工作台集中处理，从业务页面就近完成动作
- 提醒状态自动跟随业务状态变化，无需额外维护一套独立消息生命周期
- 支持“稍后处理”能力，允许用户将提醒延后到固定时间再次出现

**第一期不追求**：
- 完整站内信/消息中心
- 团队维度提醒
- 历史归档、批量处理、已读体系、实时推送

---

## 2. 范围定义

### 2.1 提醒对象

第一期仅面向**本人相关事项**，不包含主管查看团队成员、下属日报、团队线索等团队视角提醒。

### 2.2 第一期开启的提醒类型

1. 日报未提交提醒
2. 日报被评论提醒
3. 日报被 @ 提醒
4. 待我审批提醒
5. 审批结果提醒（通过 / 驳回）
6. 合同到期提醒
7. 线索超时未跟进提醒

### 2.3 第一阶段明确不做

- 独立消息中心页面
- 消息分类管理
- 标记已读 / 未读体系
- 批量处理
- 历史归档
- 实时推送（如 WebSocket）
- 团队视角提醒
- 自定义提醒订阅配置

---

## 3. 信息架构与展示承载

第一期以统一的“我的提醒”数据源为核心，但按使用场景拆分为三层展示：

### 3.1 顶部铃铛

放置在 `MainLayout` 顶部导航，承担全局感知入口。

职责：
- 显示当前未处理提醒总数
- 展示最近且最重要的提醒摘要
- 提供直接跳转业务处理入口
- 提供进入工作台“待我处理”区域的入口

### 3.2 工作台“待我处理”区域

放置在 `Dashboard` 页面，承担第一期主处理入口。

职责：
- 集中展示所有当前待处理提醒
- 按优先级和时效排序
- 提供主操作按钮
- 作为“组合式提醒体系”的核心操作面板

### 3.3 业务页面局部提示

在强相关业务入口或页面中保留局部提示，承担“就近行动”职责。

第一期建议承载位置：
- 日报入口：继续保留未提交日报提示
- 我的线索页 / 线索详情页：提示超时未跟进事项
- 其他类型先通过铃铛和工作台聚合承载，不单独扩散更多局部入口

### 3.4 为什么不是完整消息中心

本次需求核心是“待办推进”而不是“消息阅读”。因此第一期设计重点是统一提醒层，而不是建设单独的消息产品。用户看到的是提醒与任务，不是聊天式站内信列表。

## 4. 统一提醒数据模型

第一期在前端定义统一提醒条目结构，所有业务类型都通过该结构映射后再展示。

```typescript
interface ReminderItem {
  id: string;
  type: ReminderType;
  title: string;
  content?: string;
  sourceId: string;
  sourceType: 'daily_report' | 'approval' | 'contract' | 'lead';
  priority: 'high' | 'medium' | 'low';
  createdAt: string;
  deadline?: string;
  snoozedUntil?: string;
  actionLabel: string;
  actionTarget: {
    kind: 'route' | 'modal';
    path?: string;
    modal?: 'daily-report';
    params?: Record<string, string>;
  };
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| id | 提醒唯一标识 |
| type | 提醒类型，用于分类和规则判断 |
| title | 提醒标题，如“您的日报还未提交” |
| content | 可选摘要，如评论内容预览 |
| sourceId | 业务实体 ID |
| sourceType | 来源业务类型 |
| priority | 固定优先级 |
| createdAt | 提醒生成时间 |
| deadline | 截止时间或触发判断时间 |
| snoozedUntil | 稍后处理后的恢复展示时间 |
| actionLabel | 主操作按钮文案 |
| actionTarget | 点击后的行为定义 |

### 4.1 ReminderType 枚举

```typescript
type ReminderType =
  | 'daily_report_unsubmitted'
  | 'daily_report_comment'
  | 'daily_report_mention'
  | 'approval_pending'
  | 'approval_result'
  | 'contract_expiring'
  | 'lead_followup_overdue';
```

## 5. 各业务提醒映射规则

| 类型 | 触发条件 | 主操作 | 跳转目标 | 自动消失条件 |
|------|---------|--------|---------|-------------|
| 日报未提交 | 当日 18:00 后仍未提交当日日报 | 去提交 | 打开日报填写弹窗 | 提交日报 |
| 日报评论 | 我的日报收到新评论 | 查看评论 | 跳转日报详情并定位评论区 | 评论不再满足“新评论”条件时消失 |
| 日报 @ | 我的日报评论中有人 @ 我 | 查看 @ 内容 | 跳转日报详情并高亮对应评论 | @ 提醒不再满足触发条件时消失 |
| 待我审批 | 有审批单当前节点等待我处理 | 去审批 | 跳转审批详情页 | 审批通过 / 驳回 |
| 审批结果 | 我发起的审批被处理 | 查看结果 | 跳转审批详情页 | 结果提醒不再满足触发条件时消失 |
| 合同到期 | 合同在到期前 N 天内 | 查看合同 | 跳转合同详情页 | 合同续约、关闭或已不在提醒窗口 |
| 线索超时未跟进 | 到达预约跟进时间仍无新跟进，或无预约时超过固定时效仍未跟进 | 去跟进 | 跳转线索详情页 | 新增跟进记录 |

### 5.1 线索超时未跟进规则

线索提醒采用“**预约时间优先，固定时效兜底**”的判定方式：

```typescript
function isLeadOverdue(lead: Lead): boolean {
  if (lead.nextFollowupTime) {
    return new Date(lead.nextFollowupTime) < new Date();
  }

  if (lead.lastFollowupAt) {
    return Date.now() - new Date(lead.lastFollowupAt).getTime() > 48 * 60 * 60 * 1000;
  }

  if (lead.assignedAt) {
    return Date.now() - new Date(lead.assignedAt).getTime() > 48 * 60 * 60 * 1000;
  }

  return false;
}
```

规则解释：
- 若线索设置了下次跟进时间，则以该预约时间为准
- 若没有预约时间，则以最近一次跟进时间为基准，超过 48 小时未跟进则提醒
- 若从未跟进，则以分配时间为基准，超过 48 小时未跟进则提醒

### 5.2 合同到期提醒窗口

第一期合同到期提醒采用 mock 配置的固定窗口值 `N` 天，例如 7 天。到期日前进入提醒窗口后展示提醒；续约、关闭或超出该窗口后自动消失。窗口值在第一期可先写在前端配置中，不做系统配置页联动。

## 6. 组件结构

第一期建议拆分为 4 个清晰单元。

### 6.1 ReminderProvider

全局提醒状态容器，放在应用高层。

职责：
- 从各业务模块读取数据
- 调用各业务 adapter 生成统一提醒条目
- 过滤已失效提醒和稍后处理中的提醒
- 输出全量提醒、分组结果、未处理数量和操作方法

输出能力建议包括：
- `reminders`
- `groupedReminders`
- `pendingCount`
- `snoozeReminder(id, option)`
- `refreshReminders()`

### 6.2 ReminderBell

顶部铃铛组件，挂载到 `MainLayout`。

职责：
- 展示未处理数量
- 展示最近 5 条摘要
- 每条支持主操作按钮
- 每条支持“稍后处理”快捷菜单
- 提供进入工作台“待我处理”区域的入口

### 6.3 TodoPanel

工作台主待办面板，挂载到 `Dashboard`。

职责：
- 展示完整提醒列表
- 支持按类型分组显示
- 显示标题、摘要、优先级、截止时间、操作按钮
- 支持“稍后处理”快捷操作

### 6.4 ReminderSource Adapters

每个业务类型一个轻量映射函数：

- `getDailyReportReminders(...)`
- `getApprovalReminders(...)`
- `getContractReminders(...)`
- `getLeadReminders(...)`

职责：
- 从业务数据提取提醒条件
- 映射统一 `ReminderItem`
- 避免把提醒逻辑堆在 `MainLayout` 或 `Dashboard`

## 7. 数据流设计

第一期采用**前端聚合、按业务状态重算**的数据流，而不是建设独立消息表。

### 7.1 基本流程

1. `ReminderProvider` 从现有业务 mock/state 取数
2. 各 adapter 输出统一 `ReminderItem[]`
3. Provider 合并提醒并按规则排序
4. Provider 过滤：
   - 已失效项
   - `snoozedUntil > now` 的稍后处理项
5. `MainLayout` 与 `Dashboard` 读取同一份提醒结果
6. 用户完成业务动作后，业务状态变化，Provider 重新计算，提醒自动消失

### 7.2 为什么采用“重算”而非“手工删除”

本次提醒大多存在明确业务终态：
- 日报提交后，未提交提醒自然失效
- 审批完成后，待审批提醒自然失效
- 新增跟进后，线索超时提醒自然失效
- 合同续约或关闭后，到期提醒自然失效

因此第一期不维护单独“是否已处理”的提醒真相源，而是以业务状态作为唯一真相源，提醒层只负责派生展示。

## 8. 稍后处理设计

第一期支持“稍后处理”，但只支持固定时间快捷选项，不支持用户自由选择日期时间。

### 8.1 快捷选项

每条提醒支持以下固定选项：
- 1 小时后
- 今天下班前（18:00）
- 明天上午（09:00）

### 8.2 交互规则

- 点击“稍后处理”后，当前提醒立即从当前列表中隐藏
- 到达 `snoozedUntil` 时间后重新参与提醒计算并重新展示
- 若业务在稍后期间已被处理，则提醒不再恢复显示
- 稍后处理只影响展示时机，不改变业务状态

### 8.3 第一阶段存储策略

第一期可先采用前端内存 / 本地 mock 状态保存 `snoozedUntil`，不要求跨浏览器刷新长期持久化。若页面刷新后稍后状态丢失，仍可接受；后续若进入真实接口阶段，再扩展持久化策略。

## 9. 排序与优先级

### 9.1 固定优先级

**High**
- 待我审批
- 日报未提交
- 线索超时未跟进

**Medium**
- 合同到期
- 审批结果

**Low**
- 日报评论
- 日报 @

### 9.2 排序规则

统一按以下顺序排序：
1. 优先级
2. 是否临近截止 / 是否已逾期
3. 创建时间倒序

### 9.3 不同展示面的数量策略

- 顶部铃铛：显示最近且最重要的 5 条
- 工作台待办区：显示完整提醒列表
- 局部提示：只显示与当前业务直接相关的一条提醒或计数摘要

## 10. 交互细节

### 10.1 顶部铃铛

交互能力：
- 显示总未处理数量
- 展示提醒摘要（标题 + 简要说明 + 时间）
- 每条支持主操作按钮
- 每条支持稍后处理菜单
- 提供“查看全部待我处理”入口，定位到工作台待办区域

### 10.2 工作台待我处理区

展示方式建议按类型分组：
- 日报
- 审批
- 客户 / 合同
- 线索 / 任务

每条卡片展示：
- 标题
- 摘要信息
- 优先级标记
- 截止时间 / 逾期提示
- 主操作按钮
- 稍后处理按钮

### 10.3 业务页局部提示

只承担“就近行动”而不承担复杂管理：
- 日报入口保留未提交标记
- 我的线索页 / 线索详情页提示超时未跟进数量或当前条目
- 点击后应直接进入对应处理动作或详情页

## 11. 异常与边界处理

第一期采用静默降级策略：

- 某类业务数据不存在：该类提醒不生成
- 某条提醒的详情页不存在：点击回退到对应业务列表页
- 多条同类提醒同时出现：按条展示，不先做复杂聚合
- 同一业务触发多个不同提醒条件：允许并存，只避免完全重复条目
- 时间判断统一基于前端当前时间，不先引入服务端时区同步复杂度

### 11.1 关于“新评论 / 新@ / 审批结果”的处理口径

由于第一期不做已读体系，这类提醒的“消失”条件不依赖用户手动标记已读，而依赖 mock 规则中的“仍满足提醒触发条件”判断。也就是说，第一期应在 mock 数据层明确哪些评论 / @ / 审批结果属于当前活跃提醒，提醒层只消费该结果，不自行发明额外已读机制。

## 12. 与现有代码结构的衔接

### 12.1 现状

当前 `MainLayout.tsx` 已包含：
- 日报未提交状态判断
- 顶部日报图标提醒
- 静态通知铃铛红点

但这些状态仅服务于日报相关逻辑，且分散在页面内局部 state 中，不能复用于审批、合同、线索等流程。

### 12.2 第一阶段改造方向

- 保留 `MainLayout` 作为顶部全局入口承载层
- 将现有日报提醒逻辑从局部 state 迁移到统一 `ReminderProvider`
- 将静态通知铃铛改造成真实提醒入口
- 在 `Dashboard` 新增“待我处理”区块
- 为各业务模块增加 adapter，而不是把所有提醒判断直接写进布局组件

这样可以在不重构全应用的前提下，为后续新增提醒类型预留统一扩展路径。

## 13. 验证方式

第一期以前端 mock 数据验证为主，不依赖真实后端接口。

验证内容：
- 提醒条数是否正确
- 排序是否符合优先级和时效规则
- 铃铛摘要与工作台展示是否一致
- 稍后处理后是否正确隐藏与恢复
- 业务动作完成后提醒是否自动消失
- 跳转目标是否正确可达

验证手段：
- 在 `ReminderProvider` 中构造各类 mock 数据
- 通过 UI 交互直接验证展示和跳转
- 通过模拟业务动作验证提醒重算与自动消失

## 15. 实施边界总结

第一期本质上是一个**统一的流程型待办提醒层**，而不是完整站内消息系统。

其核心特征是：
- 统一入口
- 统一提醒模型
- 按业务状态实时派生
- 支持固定时间的稍后处理
- 以工作台集中处理和业务就近处理为主

在这个基础上，后续若需要扩展为完整消息中心、团队提醒、真实后端消息表或实时推送，可以继续在统一模型之上逐步演进，而不需要推翻第一期结构。
