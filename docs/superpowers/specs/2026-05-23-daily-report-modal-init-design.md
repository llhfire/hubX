# 日报模态框初始化稳定性设计

- 日期：2026-05-23
- 主题：修复点击日报按钮后，日报模态框首次打开时先显示空态、随后才渲染示例数据的问题
- 范围：仅修复日报弹窗首次显示时的初始化时序，不扩展到提醒系统、角色选择弹窗或模板体系重构

## 背景

当前顶部日报入口位于 `src/app/components/MainLayout.tsx`。用户点击日报按钮后，先打开角色选择弹窗；选择“销售日报”或“通用日报”后，再显示 `DailyReportModal`。

现象是：角色确认后，日报模态框首次显示时会先出现未渲染/空态，随后才补出该角色对应的示例数据。用户感知为弹窗在“未渲染示例数据”和“已渲染示例数据”两个状态之间切换。

## 目标

修复后，日报模态框在首次可见的第一帧就应稳定显示当前角色对应的默认内容，不出现先空白后补数据的闪烁。

具体目标如下：

1. 角色一旦确认，日报模态框首次显示时就拿到最终角色。
2. 模态框首次可见时就具备该角色对应的默认表单内容。
3. 打开过程中不出现中间空态或延迟补数据的视觉切换。
4. 不改变日报提交流程。
5. 不改变角色选择流程。
6. 不修改日报示例数据内容本身。

## 现状与根因判断

### 入口时序

`MainLayout` 当前流程为：

1. 点击日报按钮，打开 `RoleSelectModal`
2. 角色选定后：
   - `setSelectedRole(role)`
   - 关闭角色选择弹窗
   - 通过 `setTimeout(..., 50)` 延迟打开 `DailyReportModal`

该实现依赖固定时间延迟来等待状态稳定，属于时间驱动而非状态驱动。

### 弹窗内部状态

`DailyReportModal` 当前持有：

- `reportDate`
- `content`
- `isFirstRender`

其中：

- 关闭弹窗时会把 `isFirstRender` 重置为 `true`
- 关闭弹窗时会把 `content` 清空为 `null`
- 模板组件接收 `initialContent={content as ... | undefined}`

因此，弹窗首次显示时很可能先以空 `content` 进入渲染，再由模板内部逻辑补齐示例数据，形成“先空态、后有示例数据”的闪烁。

### 根因结论

根因不是模板内容错误，而是**模态框的首次显示时机早于默认内容准备完成**，并且入口依赖 `setTimeout` 来规避状态时序问题，导致首帧无法保证稳定。

## 设计原则

1. **状态驱动，不靠时间驱动**：不再依赖 `setTimeout(50)` 之类的延时让界面“碰运气”稳定。
2. **首次可见即完整**：日报模态框第一次显示时，角色和初始内容都必须已经准备好。
3. **只在打开时初始化一次**：避免在弹窗打开后因为额外状态变化再次覆盖用户编辑内容。
4. **最小改动**：优先只修改 `MainLayout` 与 `DailyReportModal`，不扩散到日报模板体系。

## 方案选择

本次采用的方案是：**在日报模态框显示前完成角色与默认内容准备，并让弹窗只在打开事件发生时执行一次初始化。**

未采用的备选方案：

- 强制按角色重新挂载整个模态框实例：可行但偏硬，超出本次最小改动诉求。
- 重构日报模板与内部状态管理：收益不足，不符合本次范围控制。

## 组件级设计

### 1. `MainLayout` 设计

文件：`src/app/components/MainLayout.tsx`

#### 调整目标

把“角色选择完成后如何打开日报弹窗”的逻辑从时间驱动改成状态驱动。

#### 设计要求

1. 角色确认后，立即确定这次打开所使用的角色。
2. 只有在角色已准备好的前提下，才允许日报模态框进入可见状态。
3. 移除 `setTimeout(..., 50)` 这类基于时间的打开方式。

#### 结果预期

`DailyReportModal` 第一次进入渲染时，`defaultRole` 已经是这次用户刚选择的角色，而不是上一轮残留值或过渡态。

### 2. `DailyReportModal` 设计

文件：`src/app/pages/daily-report/DailyReportModal.tsx`

#### 调整目标

保证弹窗在每次“从关闭到打开”的瞬间，就同步建立该角色的初始内容，并在首帧渲染完整默认数据。

#### 设计要求

1. 初始化应绑定到“本次打开事件”，而不是普通渲染过程。
2. 初始化内容来源于当前 `defaultRole`。
3. 首次可见时，模板组件拿到的就是该角色对应的默认内容，而不是 `null`/`undefined` 空值。
4. 用户开始编辑后，不再因为额外 effect 或角色相关状态变化而重置输入。
5. 关闭弹窗后，下次重新打开才重新初始化。

#### 状态策略

- `content` 不再作为“先空着，等模板自己补”的占位态使用。
- 打开弹窗时，应先同步决定本次内容初值，再渲染模板。
- 若内部保留“首次打开”判定，应确保它真正服务于“只在打开时初始化一次”，而不是留下无效或重复的状态位。

## 数据流设计

目标数据流如下：

1. 用户点击日报按钮。
2. 打开角色选择弹窗。
3. 用户选择角色。
4. 主入口先确认本次角色。
5. 日报模态框根据该角色建立本次初始内容。
6. 日报模态框首次显示时直接渲染完整默认内容。
7. 用户编辑、提交或关闭。
8. 关闭后清理本次会话态；下一次打开再按新角色重新初始化。

该数据流要求“角色确认”先于“模态框可见”，并要求“初始内容建立”先于“模板首次展示”。

## 错误与边界处理

本次不新增新的用户可见错误提示，但需要保证以下边界：

1. **销售日报** 打开时只渲染销售默认内容。
2. **通用日报** 打开时只渲染通用默认内容。
3. 从销售关闭后再打开通用，不能残留销售内容。
4. 从通用关闭后再打开销售，不能残留通用内容。
5. 用户已开始编辑后，不应因内部再次初始化而丢失输入。

## 风险点

### 风险 1：用户编辑内容被覆盖

如果初始化逻辑绑定到 `visible` 或 `defaultRole` 的一般性变化，而不是“本次打开开始”，则可能在用户输入过程中再次重置表单。

**应对：** 初始化只允许在“关闭 → 打开”的边界执行一次。

### 风险 2：上次角色内容残留

如果销售与通用共享同一份内部状态，但本次打开前没有正确建立新初值，就会把上一次内容带进来。

**应对：** 每次打开必须基于当前角色重新建立一次本次内容初值。

### 风险 3：继续保留时间延迟掩盖问题

若入口仍使用 `setTimeout`，则问题只是概率下降而非真正消除。

**应对：** 移除延时打开逻辑，改为显式状态驱动。

## 验收标准

修复完成后应满足：

1. 点击顶部日报按钮并选择“销售日报”后，模态框首帧即显示销售默认内容。
2. 点击顶部日报按钮并选择“通用日报”后，模态框首帧即显示通用默认内容。
3. 打开过程中不出现“先空白后补数据”的闪烁。
4. 关闭再重新打开，表现依旧稳定。
5. 手动编辑字段后，不会在弹窗保持打开时被重新初始化覆盖。
6. 提交日报功能保持原有行为不变。

## 运行时验证计划

验证以真实界面行为为准：

1. 启动本地前端。
2. 点击顶部日报按钮。
3. 选择“销售日报”，观察首帧内容是否直接完整显示。
4. 关闭弹窗后重新打开。
5. 选择“通用日报”，观察首帧内容是否直接完整显示。
6. 在打开状态下编辑若干字段，确认输入不会被再次初始化覆盖。
7. 完成一次提交，确认提交流程正常。

## 本次不做的事

为保持最小改动，本次明确不做：

- 重构日报模板结构
- 抽离新的日报初始化 hook 或 store
- 调整 ReminderBell 或提醒系统逻辑
- 改动角色选择弹窗的交互样式
- 改写日报示例数据内容

## 结论

本次修复的本质，是把日报模态框的初始化从“打开后补数据”改成“显示前准备完成”。

只要做到以下两点，问题就会消失：

1. `MainLayout` 不再通过时间延迟打开日报弹窗，而是在角色确认后、状态准备完成后再显示。
2. `DailyReportModal` 在每次打开时同步建立当前角色的初始内容，并确保首帧即渲染完整默认数据。
