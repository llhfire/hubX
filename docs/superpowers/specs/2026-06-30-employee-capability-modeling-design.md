# 员工能力数值建模 — 设计规格

## 概述

将 RPG 角色成长系统引入企业管理，对员工进行数值化能力建模。覆盖从招聘面试到离职的完整生命周期，通过项目实践积累经验、解锁技能、逐步晋级，为晋升评审提供客观数据支撑。

## 一、核心数据模型

### 1.1 五维能力值（0-100）

| 维度 | 标识 | 含义 |
|------|------|------|
| 技术能力 | `tech` | 编程、架构、调试、代码质量 |
| 业务能力 | `biz` | 需求理解、客户沟通、方案设计 |
| 管理能力 | `mgmt` | 项目管控、团队协调、进度把控 |
| 工具熟练 | `tool` | Figma、JIRA、框架、平台工具等 |
| 领域知识 | `domain` | 电商、金融、医疗等行业经验 |

### 1.2 等级体系（L1-L10）

每个职位有独立的等级阶梯：

```
L1 实习生 → L2 初级 → L3-L4 中级 → L5-L6 高级 → L7-L8 专家 → L9-L10 架构/总监
```

### 1.3 职位权重映射

不同职位对五维能力值有不同的权重，加权总分 = Σ(能力值 × 职位权重)。

| 职位 | 技术 | 业务 | 管理 | 工具 | 领域 |
|------|------|------|------|------|------|
| 前端开发 | 40% | 15% | 10% | 25% | 10% |
| 后端开发 | 45% | 10% | 10% | 20% | 15% |
| 项目经理 | 15% | 30% | 35% | 10% | 10% |
| UI 设计师 | 20% | 15% | 5% | 45% | 15% |
| 投放运营 | 15% | 30% | 15% | 20% | 20% |
| 销售 | 5% | 40% | 15% | 10% | 30% |

权重可由管理员按职位自定义配置。

### 1.4 技能树

- 按技能领域组织（前端开发、DevOps、项目管理、UI/UX 等）
- 最多 3 层：基础 → 进阶 → 专家
- 解锁条件：前置技能已解锁 **且**（能力值达标 **或** 参与相关项目实践）
- 允许跨岗位学习技能（如前端开发学习项目管理技能）
- 技能状态：已解锁（显示熟练度：入门/熟练/精通）、已锁定（显示解锁条件）

## 二、经验获取机制

### 2.1 日报驱动（日常）

日报中「工作种类」映射到能力维度，自动累积经验：

| 工作种类 | 映射维度 | 每 8 小时经验 |
|---------|---------|-------------|
| 开发编码 | 技术能力 | +3 |
| 需求沟通 | 业务能力 | +2 |
| 方案编写 | 业务能力 | +2 |
| 项目管理 | 管理能力 | +2 |
| 测试验收 | 技术能力 | +2 |
| 文档编写 | 业务能力 | +1 |
| 投放优化 | 领域知识 | +3 |
| 数据分析 | 领域知识 | +2 |
| 会议 | 管理能力 | +1 |

### 2.2 项目结算（阶段性）

项目交付后按角色和参与度一次性结算：

| 项目角色 | 基础经验 | 分配示例 |
|---------|---------|---------|
| 技术负责人 | +20 | 技术+10, 管理+10 |
| 核心开发 | +15 | 技术+12, 工具+3 |
| 辅助开发 | +8 | 技术+6, 工具+2 |
| UI 设计 | +15 | 工具+12, 业务+3 |
| 项目经理 | +20 | 管理+15, 业务+5 |

项目规模系数：小型 ×0.8 / 中型 ×1.0 / 大型 ×1.5 / 战略级 ×2.0

### 2.3 主管评定（校准）

每月/每季度主管可对下属分配额外经验：

| 评定等级 | 额外经验池 |
|---------|----------|
| 超出预期 | +10~20 |
| 符合预期 | +5~10 |
| 需改进 | 0 |

主管可将经验池自由分配到各维度，不强制平均分配。

## 三、晋级流程

```
数值达标 → 解锁晋级资格 → 评审委员会审批 → 晋升
```

1. **数值达标**：加权总分达到下一级门槛，系统自动检测
2. **解锁资格**：通知员工本人 + 直属主管
3. **评审审批**：评审委员会参考能力数值 + 项目表现投票决定
4. **晋升**：等级 +1，记录晋级历史，生成画像快照

晋级门槛由管理员按职位独立配置。初始参考值（前端开发）：
- L2→L3：加权总分 ≥ 25
- L4→L5：加权总分 ≥ 50
- L6→L7：加权总分 ≥ 65
- L8→L9：加权总分 ≥ 80

以上数值为初始设定，上线后根据实际数据分布调优。

## 四、员工全生命周期画像

### 4.1 三张核心画像

| 画像 | 阶段 | 内容 | 状态 |
|------|------|------|------|
| 面试画像 | 招聘阶段 | 测试得分 + 面试官结构化评分 + 历史项目 → 综合得分 + 建议定级 | 生成后存档 |
| 入职画像 | 入职时 | 面试画像 → 初始能力值 + 入职前履历 + 初始等级 | 入职时快照 |
| 在职画像 | 在职期间 | 动态更新的能力值 + 技能树 + 项目贡献 + 当前等级 | 实时动态 |

### 4.2 画像版本管理

- 面试画像：面试完成后生成，作为录用决策参考
- 入职画像：入职当天从面试画像自动转换，存档
- 晋级快照：每次晋升自动生成
- 季度快照：每季度自动生成（可配置）
- 手动快照：管理者可随时手动创建
- 离职画像：离职时生成最终存档

所有画像版本可列表查看、对比（展示成长轨迹）。

## 五、履历管理

### 5.1 入职前履历

- 手动录入：公司名称、时间段、职位、技术栈、项目职责
- 面试阶段由 HR 或面试官录入
- 入职前项目经验自动转化为初始能力值

### 5.2 在职履历

- 系统自动从项目管理模块同步（项目名称、角色、工时、日期）
- 支持手动补充（内部培训、技术分享等非项目活动）
- 每条记录关联经验获得量

## 六、招聘量化模型

### 6.1 面试评分卡

**客观测试（自动评分）：**
- 在线编程测试（0-100）
- 算法逻辑题（0-100）
- 项目设计题（0-100）

**面试官结构化评分（1-10）：**
- 技术深度
- 沟通表达
- 项目经验
- 学习潜力
- 团队协作
- 抗压能力

### 6.2 面试画像生成规则

客观测试得分 + 面试官评分 → 五维能力初始值：
- 编程测试 → 技术能力（映射公式：得分 × 0.5）
- 项目设计 → 业务能力（得分 × 0.25）
- 技术深度 → 工具熟练（评分 × 5）
- 沟通表达 → 管理能力（评分 × 1）
- 历史项目 → 领域知识（按项目数量和复杂度）

综合得分 = Σ(初始能力值 × 职位权重)，用于建议定级。

## 七、与现有/规划模块的关联

| 模块 | 关联方式 |
|------|---------|
| 员工管理 | 员工基础信息（职级/部门/入职日期）是能力建模的底座 |
| 日报模块 | 工作种类映射能力维度，日常经验自动累积 |
| 项目管理 | 项目结算发放经验，参与项目触发技能解锁 |
| 项目成本核算 | 员工等级 → 职级标准时薪 → 人力成本自动计算 |
| 个人工作台 | 展示个人能力面板、等级进度、下一个可解锁技能 |

## 八、UI 布局

采用**纵向详情页**布局（方案 B）：
- 左侧栏：头像 + 能力进度条 + 加权总分 + 晋级历史
- 右侧主区域：技能树（含跨岗位技能）+ 项目贡献列表
- 顶部标签切换：在职画像 / 履历 / 画像版本 / 面试画像（如有）

## 九、术语

| 术语 | 英文 | 定义 |
|------|------|------|
| 能力值 | Ability Score | 五维能力数值，0-100 |
| 加权总分 | Weighted Score | Σ(能力值 × 职位权重) |
| 技能树 | Skill Tree | 分层技能依赖关系，最多3层 |
| 经验值 | Experience Point | 日报/项目/主管评定获得的点数 |
| 面试画像 | Interview Profile | 面试阶段的客观+主观量化数据 |
| 入职画像 | Onboarding Profile | 入职时的能力初始快照 |
| 在职画像 | Active Profile | 动态更新的当前能力状态 |
| 晋级资格 | Promotion Eligibility | 加权总分达标后解锁的晋升条件 |
