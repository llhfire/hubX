# 项目详情页首屏摘要区重设计

**日期**：2026-06-13
**状态**：设计完成，待实施
**负责人**：待定
**关联文件**：`HubX/src/app/pages/ProjectDetail.tsx`

---

## 背景与目标

当前 `ProjectDetail` 顶部有 5 个卡片区域：

1. 总进度（Statistic + Progress）
2. 负责人（Statistic）
3. 预计结束日期（Statistic）
4. 已用总工时（Statistic）
5. 交付进度（`DeliveryProgressCard`，带环形图）

这 5 个框屏占比过大，且信息以静态展示为主，风险感知能力弱。用户需要往下翻才能发现项目问题。

本次重构目标是：

- 去掉"总进度"卡片。
- 把"交付进度"简化成紧凑摘要卡并纳入首行。
- 把剩余 4 个卡全部改为"风险预警导向"：每卡都承载主值 + 预警句 + 补充说明。
- 首屏从 `5 个大框` 收敛为 `1 行 4 个紧凑预警卡`。

---

## 设计

### 首行布局

保持 `4 张等宽卡片` 在同一行，不再保留第二行的独立交付进度大卡。每张卡统一改成紧凑摘要结构：

```
┌─────────────────┬─────────────────┬─────────────────┬─────────────────┐
│ [风险标签]      │ [风险标签]      │ [风险标签]      │ [风险标签]      │
│ [主值大字]      │ [主值大字]      │ [主值大字]      │ [主值大字]      │
│ [预警句]        │ [预警句]        │ [预警句]        │ [预警句]        │
│ [补充说明]      │ [补充说明]      │ [补充说明]      │ [补充说明]      │
└─────────────────┴─────────────────┴─────────────────┴─────────────────┘
```

去掉 `Statistic + Progress 大组合`、环形图和长段说明，让这一排成为"首屏预警条"，高度明显低于现状。

### 四张卡片内容

#### 1. 交付进度卡

| 层级 | 内容 |
|------|------|
| 主值 | 当前交付阶段 + 完成百分比，例如「开发中 / 65%」 |
| 预警句 | 延期最严重的工作项，例如「UI 评审逾期 6 天」 |
| 补充说明 | 下一里程碑，或「当前无逾期，最接近到期的是 XX」 |

- 数据口径：从 `initialDeliveryPlans[project.id]` 的步骤和里程碑计算；逾期判断复用现有 `isStepOverdue()` 规则。
- 风险分级：有逾期工作项即至少 `预警`，逾期天数最大的进入 `严重`。

#### 2. 负责人卡

| 层级 | 内容 |
|------|------|
| 主值 | 当前负责人姓名 |
| 预警句 | 该负责人同时负责的其他高优先级项目，例如「另有 2 个高优项目并行」 |
| 补充说明 | 列出最多 2 个项目名，超出则用「等 N 个」收口 |

- 数据口径：从 `initialProjects` 筛选「同负责人、优先级=高、非当前项目、状态未完成」的项目。
- 风险分级：并行高优项目 `>=2` 为 `预警`，`>=3` 为 `严重`。

#### 3. 交付时间卡

| 层级 | 内容 |
|------|------|
| 主值 | 剩余 X 天 / 已逾期 X 天（带正负方向） |
| 预警句 | 合同约定交付日：YYYY-MM-DD |
| 补充说明 | 风险分层文案：「时间充足」「临近交付」「已逾期」 |

- 数据口径：当前阶段用 `expectedEndDate` 作为合同交付日代理字段；后续合同模块接入后再切换。
- 风险分级：剩余 `<=7` 天为 `预警`，已逾期为 `严重`。

#### 4. 总工时卡

| 层级 | 内容 |
|------|------|
| 主值 | 累计总工时，例如「128H」 |
| 预警句 | 工时消耗最高的 3 名成员，例如「李四 32H / 王五 28H / 赵六 20H」 |
| 补充说明 | 集中度提示，例如「前三成员占比 68%」 |

- 数据口径：直接复用当前 `memberHours` 和 `totalHours`，按工时倒序取前 3。
- 风险分级：前三成员占比过高，或第一名显著高于其他人时记为 `注意/预警`。

### 统一预警样式

- 每张卡都带一个小型风险标签：`正常（绿）/ 注意（蓝）/ 预警（橙）/ 严重（红）`。
- 颜色仅用于标签和预警短句，不再做大面积色块底。
- 卡片高度统一、结构统一，便于一眼横向比较。

### 空状态与降级策略

| 场景 | 降级行为 |
|------|----------|
| 交付计划不存在 | 显示「暂未生成交付计划」，风险标签为 `注意`，补充「无法识别延期步骤」 |
| 负责人无其他高优项目 | 显示「当前无高优并行项目」，风险标签为 `正常` |
| `expectedEndDate` 为空 | 显示「未设置交付日期」，风险标签为 `注意`，不误报逾期 |
| `memberHours` 为空 | 显示 `0H`，补充文案改为「暂无工时记录」 |

## 实施范围

本次只重构 `ProjectDetail` 顶部摘要区（`Grid.Row` 首行 4 卡 + 原 `DeliveryProgressCard` 的简化版摘要卡）。

**不涉及**：

- 项目列表页 `总进度` 列
- 跟进弹窗里的 `总进度` 字段
- 交付计划详情页本身（`/projects/:id/delivery`）

---

## 验证方式

至少覆盖 4 种场景：

1. 正常项目（无风险信号）
2. 存在延期步骤
3. 无交付计划
4. 无工时 / 无预计结束日期

核心验收标准：

- 首屏从原来的 `5 个大框` 收敛为 `1 行 4 个紧凑预警卡`。
- 用户无需往下翻，就能看出：哪里有风险、谁压力大、离交付还有多久、工时集中在哪些人身上。

---

## 技术实现要点

- 复用 `initialProjects`、`initialDeliveryPlans`、`initialDailyReports` 等现有 mock 数据。
- 复用 `isStepOverdue()`、`calcOverallCompletion()`、`calcPhaseCompletion()` 等现有工具函数。
- 在 `ProjectDetail.tsx` 内部新增 4 个独立子组件或纯函数，不污染全局状态。
- 预警标签颜色用 Arco Design 的 Tag 组件实现，保持与现有 UI 风格一致。
- 日期差值计算优先在 `ProjectDetail.tsx` 内以纯函数实现，输入输出统一使用现有 `YYYY-MM-DD` 字符串格式；仅在后续出现复用需求时再抽到共享工具。
