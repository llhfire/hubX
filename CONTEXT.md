# HubX 业务术语表

## 公司业务模式

HubX 是一家**软件外包 + 自运营投放获客**的复合型公司。两条业务线实际上是同一条漏斗：

> 广告投放 → 线索 → 客户 → 合同 → 项目 → 交付 → 利润

- **外包业务**：为甲方客户提供软件定制开发、小程序、APP 等交付服务
- **自运营投放**：在百度、小红书、抖音、淘宝等平台投放广告获取线索，线索转化后签外包合同

## 关键业务概念

### 线索（Lead）
- 来源：百度推广、抖音、小红书、微信推广、其他
- 归属：公海线索（所有人可见）、我的线索（归属人可见）、垃圾线索
- 线索可关联到客户主体（公司），转化后进入合同流程

### SABC 分级（工作台占位概念）
- **分级对象统一为「线索」**，不是客户主体。客户主体本身不建 S/A/B/C 级别。
- 由线索的「意向等级 × 预估预算」二维矩阵算出 S/A/B/C 四档，金字塔分布（S 少 / C 多）。
- 包含两个占位字段：`intentLevel`（扩展现有意向等级到 高/中/低/无）、`budgetEstimated`（预估预算，¥）。
- 该分级目前仅用于工作台可视化，是前端占位概念，未接入全系统业务规则；未来接入真实数据时需重新评估分级规则。

### 提醒系统与团队动态的边界
- 个人待办（日报/审批/线索/合同）的单一事实源是 `ReminderContext`，工作台任何「待办计数」「待跟进任务」都从此派生，不再另起待办概念实体。
- 「智能进度提醒」（如「3 天未触达，去破冰」「报价已发 2 天，确认反馈」）与 SABC 分级的「S/A 级超期未触达」流失预警共享同一组占位数据，不重复建模。
- 「团队动态 Feed」（签约/出差/进入阶段/业绩达成）是 HubX 全新占位概念，仅存在于工作台 mock，未接入全系统行为埋点；未来接真实数据时需补一个团队事件流系统（属第四阶段「AI 驱动」范围的子集）。

### 线索推进漏斗（工作台占位概念）
- 漏斗五段：未触达 / 初步沟通 / 跟进中 / 促成阶段 / 已流失。
- 用独立占位字段 `leadStage` 描述，**不与现有线索 `status` 字段共用**。现有页面继续用 `status`，漏斗只读 `leadStage`。
- 「已流失」段 = 已转入垃圾线索池的死透总量；与「S/A 级线索超期未触达」的流失预警（高价值抢救名单）不重叠。

### 线索治理规则
- 跟进效果不达标 → 机制限制（如限制领取新线索数量）
- 跟进动作不规范 → 机制限制（如超时未跟进回收线索）

### 合同（Contract）
- 状态机：草稿 → 审批中 → 待寄出 → 待回寄 → 已归档 → 已作废
- 从线索/报价单创建，创建后不可修改关联
- 归档后进入项目管理阶段

### 项目（Project）
- 状态：未开始、进行中、已完成、验收中、搁置、延迟、催款中
- 关联合同、线索、日报
- 交付计划按 SOP 七大板块展开
- 项目是工作项（需求/任务/缺陷）的聚合根，所有工作项必须归属一个项目

### 工作项（Work Item）
- 工作项是项目内跟踪工作的单元，分为**需求（Requirement）、任务（Task）、缺陷（Defect）**三种独立模型，不共用统一表
- 工作项通过 `type` 字段区分（`requirement` / `task` / `defect`），但各自有独立字段和状态流
- 工作项编号在项目内递增：`REQ-001`、`TSK-001`、`BUG-001`

### 需求（Requirement）
- 描述一个需要实现的功能或能力
- 状态流：待处理 → 进行中 → 已完成，旁路：已搁置
- 可拆解为多个任务（1:N），但任务也可以独立存在不强制归属需求
- 特有字段：验收标准（acceptanceCriteria）

### 任务（Task）
- 描述一个具体执行的工作单元
- 状态流：待处理 → 进行中 → 已完成，旁路：已阻塞
- 特有字段：预计工时（estimatedHours）、实际工时（actualHours）、检查清单（checklist）
- 检查清单是任务的附属字段（`{ text, done }[]`），不支持独立子任务

### 缺陷（Defect）
- 描述一个需要修复的问题或 Bug
- 状态流：待处理 → 处理中 → 待验证 → 已关闭，旁路：已拒绝、已重开
- 特有字段：严重程度（severity：致命/严重/一般/轻微）、复现步骤（reproductionSteps）
- 缺陷必须带描述，推荐附带截图

### 工作项关联
- 工作项 → 项目：多对一（必填）
- 工作项 → SOP 步骤：多对一（可选），步骤完成时提示联动工作项状态
- 工作项 → 合同：多对一（可选）
- 工作项 → 员工（负责人/创建人）：多对一（必填）
- 需求 → 任务：1:N（可选归属）
- 缺陷 → 任务/需求：可选关联
- 工作项之间支持"关联"（不区分重复/阻塞，未来可通过 type 字段升级）
- 工作项 ↔ 日报：双向关联（日报可勾选工作项，工作项完成后推送日报草稿）

### 工作项评论
- 三种工作项共用一张评论表（通过 workItemType + workItemId 区分）
- 评论支持 @提及，触发提醒

### 工作项操作历史
- 三种工作项共用一张操作历史表
- 记录：创建、状态变更、字段编辑、分配变更
- 结构：action + field + oldValue + newValue + remark

### 日报（Daily Report）
- 按角色区分模板：销售、交付、投放、通用
- 核心字段：工作种类、关联项目、工时
- 日报工时是项目成本核算的输入

### 项目成本核算
- 人力成本 = Σ (人员工时 × 职级标准时薪)
- 项目成本 = 人力成本 + 报销成本
- 项目利润 = 合同金额 − 项目成本
- 职级时薪由财务在人工成本设置中统一维护

### 售后运维
- 项目交付后有免费维护期
- 维护期满后可签付费运维合同
- 客户工单跟踪、续费到期提醒

### 员工能力建模
- 五维能力值（0-100）：技术能力、业务能力、管理能力、工具熟练、领域知识
- 职位权重映射：不同职位对五维能力有不同的权重，加权总分 = Σ(能力值 × 权重)
- L1-L10 等级体系：实习生→初级→中级→高级→专家→架构/总监
- 技能树：最多3层，前置依赖+项目实践触发解锁，允许跨岗位
- 经验三通道：日报驱动（日常）+ 项目结算（阶段性）+ 主管评定（校准）
- 晋级流程：数值达标→解锁资格→评审审批→晋升
- 三张画像：面试画像→入职画像→在职画像，含版本管理
- 招聘量化：面试评分卡（客观测试+主观评分）→ 面试画像 → 初始能力值

### 全链路 ROI
- 广告消耗 → 线索数量/成本 → 客户转化率 → 合同金额 → 项目利润
- 按渠道（百度/小红书/抖音等）拆分 ROI
